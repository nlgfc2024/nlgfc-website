# Incident Note: 504 Gateway Timeout on `/nlgfc-website/`

- Date: 2026-04-10
- Environment: Production
- Public URL: `http://41.77.13.188:8080/nlgfc-website/`
- Error: `504 Gateway Time-out` (`nginx/1.24.0`)

## Symptoms

- Page routes under `/nlgfc-website/*` timed out after ~60s.
- API routes such as `/api/contact`, `/api/vacancies`, and `/api/projects/menu` responded with `200`.

## Root Cause

- SSR page rendering depended on API calls that could block long enough for nginx upstream timeout.
- Multiple code paths defaulted API base to `http://localhost:8000` when runtime env vars were missing.
- Navbar SSR requests were not guarded with fast fail/fallback behavior.

## Changes Implemented

## 1) Runtime config default hardening

- File: `nuxt.config.ts`
- Change:
  - `baseUrl` fallback changed from `http://localhost:8000` to empty string `''`
  - `apiBase` fallback changed from `http://localhost:8000` to empty string `''`

## 2) API fetch timeout/fallback hardening

- Files:
  - `composables/useApiData.ts`
  - `composables/useDonorProjects.ts`
  - `composables/usePosts.ts`
  - `composables/usePartners.ts`
  - `components/ImageSlider.vue`
- Change:
  - Added bounded fetch behavior (`timeout`, `retry: 0`) for critical API requests.
  - Normalized API base resolution to `config.public.apiBase || config.public.baseUrl || ''`.

## 3) SSR navbar resilience

- File: `components/Navbar.vue`
- Change:
  - Added `fetchWithFallback()` helper with timeout and safe fallback payload.
  - Wrapped vacancies/procurement/project menu API calls with fallback behavior so SSR does not block route response.

## 4) Consistent API base usage in pages

- Files:
  - `pages/index.vue`
  - `pages/contact.vue`
  - `pages/about.vue`
  - `pages/news.vue`
  - `pages/resourceCenter.vue`
  - `pages/resourceCenter copy.vue`
  - `pages/localAuthorities/alldistricts.vue`
  - `composables/useLAReports.ts`
- Change:
  - Removed hardcoded localhost fallback and aligned all to runtime-safe base resolution.

## 5) Deployment workflow env propagation

- File: `.github/workflows/deploy-production.yml`
- Change:
  - Added deploy step env wiring for:
    - `NUXT_PUBLIC_BASE_URL`
    - `NUXT_PUBLIC_API_BASE`
    - `NUXT_PUBLIC_JOBS_PORTAL_BASE`
  - Exported these vars on the remote server before build/restart.

## Validation

- Local build result: `npm run build` passed after changes.
- Required production verification after deploy:
  - `curl -s -o /dev/null -w "%{http_code} %{time_total}\n" http://41.77.13.188:8080/nlgfc-website/`
  - Expect `200` and normal response time (not ~60s timeout).

## Follow-up Findings (2026-04-10)

- Smoke check output confirmed a second issue pattern:
  - `node-local` was `200` but with very high latency (~`104s`) on `/nlgfc-website/`.
  - nginx route returned `504` after ~`60s`.
- PM2 and runtime diagnostics showed:
  - `EADDRINUSE` on `127.0.0.1:3000`.
  - old/legacy process behavior repeatedly launching `nuxt preview`.
  - intermittent restart loops causing unstable upstream availability.

## Follow-up Changes Implemented

## 6) PM2 startup hardening in production workflow

- File: `.github/workflows/deploy-production.yml`
- Change:
  - Ensure old PM2 app process is removed before restart (`pm2 delete <app>`).
  - Remove known legacy process (`frontendclear`) when present.
  - Start from production output directly:
    - `pm2 start .output/server/index.mjs --name <app> --cwd <deploy_path> --update-env`
  - Export and pin runtime process env for PM2 startup:
    - `NODE_ENV=production`
    - `HOST=127.0.0.1`
    - `PORT=3000`
  - Add readiness loop against `http://127.0.0.1:3000/nlgfc-website/`.
  - Force `NITRO_PRESET=node-server` before `npm run build` to avoid accidental preview/static preset deploy behavior.
  - Install full build dependencies with `npm ci --include=dev` and defer `NODE_ENV=production` until runtime process startup.
  - Kill orphan listeners on `:3000` before PM2 startup to prevent `EADDRINUSE` from detached legacy child processes.
  - Fail deployment if PM2 script path is not `.output/server/index.mjs`.

## 7) Smoke-check diagnostics expansion

- File: `.github/workflows/deploy-production.yml`
- Change:
  - On smoke failure, collect:
    - `pm2 status`
    - `pm2 show <app>`
    - `pm2 logs --nostream --lines 200`
    - `ss -ltnp | grep ':3000|:8080'`
    - `lsof -nP -iTCP:3000 -sTCP:LISTEN`
    - `lsof -nP -iTCP:8080 -sTCP:LISTEN`
    - nginx access/error log tails
  - This captures port conflicts, restart loops, and upstream timeout causes in one run.

## 8) Homepage SSR timeout mitigation

- Files:
  - `pages/index.vue`
  - `components/ImageSlider.vue`
  - `composables/useDonorProjects.ts`
- Change:
  - Converted homepage-heavy API fetches to client-side (`server: false`) with lazy loading.
  - Reduced fetch timeout/retry windows for non-critical blocks.
  - Added safe fallback behavior for donor project fetching to prevent hard SSR blocking.
  - Goal: ensure first response is fast enough to stay below nginx upstream timeout.
